# Visual Product Search Engine

A query-by-image fashion retrieval system built on **YOLO + BLIP-2 + CLIP + HNSW**.  
Built for the *Visual Recognition Course Project* on the **DeepFashion In-Shop Clothes Retrieval** dataset.

## Project Overview

Users often find a product image online but cannot locate it in an e-commerce catalogue using text search. This system solves that by accepting an image query and returning the most visually and semantically similar products from a pre-built index.

**Offline phase**: every catalogue image is detected (YOLO), captioned (BLIP-2), encoded (CLIP), and stored in an ANN index (HNSW).  
**Online phase**: a query image is detected, encoded, searched against the index, and optionally re-ranked (BLIP-2 ITM).

```
     ▼

└──────────┘    └──────────────┘    └─────────────┬──────────────────┘
                                                   │

                                          └──────────────────┘

────────────────
Query image ──▶ YOLO crop ──▶ CLIP encode ──▶ HNSW search ──▶ BLIP-2 ITM re-rank ──▶ Top-K
```

## Design Decisions

### 1. Why `open_clip` instead of OpenAI's `clip` package?

`open_clip` (https://github.com/mlfoundations/open_clip) is a community re-implementation of CLIP that:
- Exposes native PyTorch training (no monkey-patching required).
- Supports `model.save_pretrained()` / `model.load_pretrained()` patterns.
- Provides more pretrained weight variants (ViT-L, ViT-H, etc.) under the same API.
- Is actively maintained and compatible with the latest PyTorch.

### 2. Why YOLO for detection?

- YOLO (specifically YOLOv8 via `ultralytics`) is the fastest single-shot detector available.
- The spec is model-agnostic — any YOLO variant works, so we default to the lightweight `yolov8n` (nano) for CPU compatibility. Larger variants (yolov8s, yolov8m) can be used by changing `YOLO_MODEL_NAME` in `config.py`.
- Bounding-box cropping reduces background noise that would otherwise dominate CLIP embeddings on cluttered images.
- **Fallback**: if no detection is found (e.g., studio shots with no visible clothing), the full image is used. This avoids hard failures.

### 3. Why BLIP-2 for captioning?

- BLIP-2 produces semantically rich, consistent descriptions (colour, fit, material, style) that textual search systems often miss or inconsistently tag.
- Captions bridge modalities: the same item described differently by two sellers still produces similar captions.
- BLIP-2's **Image-Text Matching (ITM)** head is reused for re-ranking, providing cross-modal reasoning beyond cosine similarity.
- BLIP-2 is kept **frozen** throughout (per spec): it is a large model (~3 B params) and domain adaptation is not needed — its pretraining covers general visual descriptions adequately.

### 4. Why fine-tune only CLIP's vision encoder?

- CLIP's **text encoder** encodes captions generated by BLIP-2; its language understanding is already well-suited to natural-language descriptions. Freezing it also reduces GPU memory.
- CLIP's **vision encoder** must adapt to the fashion domain (fabric textures, garment shapes, clothing categories that were not CLIP's primary training focus). We unfreeze only the last 4 transformer blocks to preserve general low-level features while adapting high-level semantics.
- This partial freeze prevents **catastrophic forgetting** — a common failure mode when fine-tuning large vision-language models on small domain datasets.

### 5. Why InfoNCE (contrastive) loss?

- InfoNCE (NT-Xent) is the standard loss for metric learning when you have identity supervision (same `item_id` = positive pair).
- It jointly pushes same-item embeddings together and different-item embeddings apart, directly optimising the cosine similarity structure used at retrieval time.
- Temperature (`τ = 0.07`) controls the sharpness of the distribution: lower values emphasise hard negatives (items that are visually similar but different products).

### 6. Why HNSW (Hierarchical Navigable Small World) for ANN?

- HNSW is a graph-based ANN algorithm with sub-linear query time O(log N) and excellent recall/speed tradeoffs.
- We use `hnswlib` (C++ binding): it is faster than FAISS on CPU for single-query lookup and requires no GPU for inference.
- `space="ip"` (inner product) on L2-normalised vectors equals cosine similarity — no extra normalisation at query time.
- **M=32** and **ef_construction=200** are chosen for high recall (>99 % at K=10 on 100 K vectors) with moderate build time. Reduce M to 16 for speed, increase to 64 for higher recall.

### 7. Why separate index and metadata files?

- `hnswlib` stores only float vectors internally. Metadata (item_id, path, caption) is stored in a Python dict pickled alongside the index. This keeps the index binary compact and avoids hnswlib's label API limitations.
- The two files are always saved together and loaded together via `HNSWIndex.save()` / `HNSWIndex.load()`.

### 8. Why `PairBatchSampler` for training?

- Contrastive loss requires positive pairs. `PairBatchSampler` guarantees that every batch contains exactly B anchors and B positives, each pair sharing an `item_id`.
- This is more memory-efficient than mining from a full N×N similarity matrix and works with arbitrary batch sizes.

### 9. Cosine annealing with warm-up

- Warm-up (5 % of total steps) prevents early large gradient steps from destroying the pre-trained CLIP features.
- Cosine annealing smoothly decays the learning rate to near-zero, avoiding overfitting near the end of training.

### 10. Checkpoint and resume support

- Offline indexing can take hours on large datasets. `offline_indexing.py` saves a checkpoint every 500 images so the pipeline can resume after interruption without re-processing completed images.

### 11. BLIP-2 ITM re-ranking

- Raw cosine similarity retrieves visually similar items but can confuse different items with shared colours or textures.
- BLIP-2 ITM scores how well a candidate's caption describes the query image — adding cross-modal reasoning that corrects near-miss ANN errors.
- The blended score `β·cosine + (1-β)·itm` (default β=0.5) balances both signals. Setting `β=1.0` disables re-ranking.

### 12. Ablation study design

| ID | Config | Purpose |
|----|--------|---------|
| A  | α=1.0, vision-only CLIP, frozen | Baseline: no text alignment, no fine-tuning |
| B  | α∈{0.6, 0.8}, frozen CLIP + BLIP-2 | Measures captioning gain without fine-tuning |
| C  | α∈{0.6, 0.8}, fine-tuned CLIP + BLIP-2 | Isolates fine-tuning contribution |

All conditions use the same HNSW configuration. Results are reported as mean ± std over 3–4 random seeds.

## Installation

```bash
# Clone / download the project
cd visual_product_search

# Create a virtual environment (recommended)
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt
```

**GPU (recommended for BLIP-2 + fine-tuning):**  
Install PyTorch with CUDA from https://pytorch.org before running `pip install -r requirements.txt`.

**CPU-only (slower but works):**  
The code automatically falls back to CPU. Expect BLIP-2 to be slow (~10 s/image).  
For CPU-only usage, set `load_in_8bit=False` (default) and reduce `BATCH_SIZE` to 8 in `config.py`.

## Dataset Setup

### Option A: Official DeepFashion In-Shop Clothes Retrieval

1. Download from http://mmlab.ie.cuhk.edu.hk/projects/DeepFashion/InShopRetrieval.html
2. Place images under `data/deepfashion/img/`
3. Place annotation files under `data/deepfashion/Anno/`:
   - `list_eval_partition.txt`
   - `list_bbox_inshop.txt` (optional)

The code reads the official partition file automatically when `--split official` is used.

### Option B: Custom Dataset

Organise images with item-level subdirectories:
```
data/custom_dataset/
├── item_001/
│   ├── front.jpg
│   └── back.jpg
├── item_002/
│   └── photo.jpg
...
```

Then run:
```bash
python run_indexing.py --root data/custom_dataset --split scan --tag custom
```

### Option C: Pre-built split files

Create `data/splits/train.txt`, `gallery.txt`, `query.txt` with format:
```
relative/path/to/image.jpg  item_id
```

Then use `--split csv`.

## Running the Pipeline

### Step 1: Fine-tune CLIP (optional, Ablation C)

```bash
# Single seed
python run_finetune.py --seed 42 --epochs 10

# All seeds (for mean ± std reporting)
python run_finetune.py --multiseed --epochs 10
```

### Step 2: Build the Index

```bash
# Ablation A — vision only, frozen
python run_indexing.py --alpha 1.0 --no-blip2 --tag ablation_A

# Ablation B — frozen CLIP + BLIP-2
python run_indexing.py --alpha 0.6 --tag ablation_B_a06
python run_indexing.py --alpha 0.8 --tag ablation_B_a08

# Ablation C — fine-tuned CLIP + BLIP-2
python run_indexing.py --alpha 0.6 --finetuned --tag ablation_C_a06
python run_indexing.py --alpha 0.8 --finetuned --tag ablation_C_a08
```

### Step 3: Evaluate

```bash
python run_evaluation.py --split official --index-tag ablation_A --no-rerank
python run_evaluation.py --split official --index-tag ablation_B_a06
python run_evaluation.py --split official --index-tag ablation_C_a06 --finetuned
```

Results are saved to `results/<tag>_metrics.json`.

## Streamlit Demo

```bash
streamlit run streamlit_app/app.py
```

**UI flow:**
1. Upload any clothing image.
2. YOLO crops the product region (multiple detections shown for selection).
3. User confirms or selects a different crop.
4. CLIP encodes the crop; HNSW retrieves top-K matches.
5. (Optional) BLIP-2 ITM re-ranks results.
6. Results displayed as a grid with scores, item IDs, and captions.

**Sidebar controls:**
- Index tag (must match what was built)
- Fusion α weight
- Top-K slider
- BLIP-2 re-ranking toggle
- Dataset root (for showing actual product images)

## Evaluation

Metrics computed (per spec):

| Metric | Description |
|--------|-------------|
| **Recall@K** | ≥1 relevant item in top-K (binary: 0 or 1 per query) |
| **NDCG@K** | Normalised DCG — rewards placing relevant items earlier |
| **mAP@K** | Mean Average Precision — rewards both finding and ranking |

K ∈ {5, 10, 15} as specified.  
Results reported as **mean ± std** over 3–4 random seeds.

Ground truth: two images are a correct match iff they share the same `item_id`.

## Local vs Online Model Loading

Every model in the system supports **local caching** to avoid repeated downloads:

### YOLO
- **First run**: ultralytics auto-downloads `yolov8n.pt` and saves to `models/yolo/yolov8n.pt`.
- **Subsequent runs**: loaded from disk, no internet required.
- **Change model**: edit `YOLO_MODEL_NAME` in `config.py` (e.g., `"yolov8s.pt"` for higher accuracy).

### CLIP
- **First run**: `open_clip` downloads `ViT-B/32` (OpenAI weights) and saves to `models/clip_base.pt`.
- **After fine-tuning**: fine-tuned weights are saved to `models/clip_finetuned.pt` and loaded automatically on next run.
- **Epoch checkpoints**: saved to `models/clip_checkpoints/clip_epoch_NNN.pt` for recovery.

### BLIP-2
- **First run**: HuggingFace downloads `Salesforce/blip2-opt-2.7b` (~15 GB) and saves to `models/blip2/`.
- **Subsequent runs**: loaded from `models/blip2/` directory (no internet).
- **8-bit quantisation**: set `load_in_8bit=True` in `BLIP2Captioner(...)` to reduce GPU memory from ~16 GB to ~4 GB.

### Forcing local-only mode

Set environment variable to prevent HuggingFace from going online:
```bash
export TRANSFORMERS_OFFLINE=1
export HF_DATASETS_OFFLINE=1
```

## Configuration Reference

All settings are in `config.py`. Key parameters:

| Parameter | Default | Description |
|-----------|---------|-------------|
| `ALPHA` | 0.6 | Image weight in fusion (1.0 = vision only) |
| `CLIP_MODEL_NAME` | `"ViT-B/32"` | CLIP architecture |
| `CLIP_PRETRAINED` | `"openai"` | CLIP weight set |
| `BLIP2_MODEL_NAME` | `"Salesforce/blip2-opt-2.7b"` | HuggingFace model ID |
| `YOLO_MODEL_NAME` | `"yolov8n.pt"` | YOLO variant |
| `HNSW_M` | 32 | HNSW graph connections per node |
| `HNSW_EF_CONSTRUCTION` | 200 | Index build quality |
| `HNSW_EF_SEARCH` | 100 | Query-time exploration |
| `BATCH_SIZE` | 32 | Training batch size (pairs) |
| `NUM_EPOCHS` | 10 | Fine-tuning epochs |
| `LEARNING_RATE` | 1e-5 | AdamW learning rate |
| `TEMPERATURE` | 0.07 | InfoNCE temperature |
| `TRAIN_LAST_N_BLOCKS` | 4 | Vision blocks to unfreeze |
| `TOP_K_VALUES` | [5,10,15] | Evaluation cutoffs |
| `SEEDS` | [42,0,7,21] | Random seeds for multi-seed runs |
